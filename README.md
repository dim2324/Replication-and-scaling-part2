# Домашнее задание к занятию "Репликация и масштабирование. Часть 2»" - `Николаев Дмитрий`


### Задание 1
Опишите основные преимущества использования масштабирования методами:
Активный master-сервер и пассивный репликационный slave-сервер;
master-сервер и несколько slave-серверов;
Дайте ответ в свободной форме.

## 1. Активный master-сервер и пассивный репликационный slave-сервер
 Все операции записи (INSERT/UPDATE/DELETE) идут только на мастер, а слейв в реальном времени (или с небольшой задержкой) воспроизводит бинарный лог мастера, оставаясь «пассивным» — то есть не принимает запросы на изменение данных.
*   *Резервирование и отказоустойчивость (Failover).* Это самое главное. Если мастер «падает» (аппаратный сбой, перегрузка, сетевой обрыв), есть горячий резерв. Слейв содержит актуальную копию данных, можно оперативно переключить на него приложение .Время простоя системы сокращается с часов до минут.
*   *Безопасность резервного копирования.* Можно делать физический бекап не на мастере, а на слейве. Это не блокирует рабочие таблицы мастера и не создает нагрузку на продакшн. Так же можно остановить репликацию на время бекапа, чтобы получить консистентный снапшот, а потом запустить ее снова.
*   *Геораспределенность.* Слейв можно разместить в другом дата-центре или регионе. Это дает два бонуса: локальные пользователи читают данные быстрее (низкая задержка).

## 2. Master-сервер и несколько slave-серверов
*   *Горизонтальное масштабирование чтения (Read Scaling).* Это главный драйвер. В реальных веб-приложениях соотношение операций чтения к записи часто составляет 10:1 или даже 100:1. Имея 3–5 слейвов, можно распределять все `SELECT`-запросы между ними через балансировщик (например, ProxySQL или HAProxy). Каждый слейв берет на себя часть нагрузки, и суммарная пропускная способность системы по чтению растет почти линейно с каждым добавленным сервером.
*   *Каскадная репликация.* При большом количестве слейвов ( 10+) можно не вешать их всех напрямую на мастер, чтобы не создавать ему лишнюю сетевую нагрузку. Вместо этого один слейв становится «релейным» (master для других слейвов), что снижает количество соединений к главному серверу.




### Задание 2
Разработайте план для выполнения горизонтального и вертикального шаринга базы данных. База данных состоит из трёх таблиц:
пользователи,книги,магазины (столбцы произвольно).
Опишите принципы построения системы и их разграничение или разбивку между базами данных.
Пришлите блоксхему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.

* Структура таблиц :
users (user_id, full_name, email, phone, city, registration_date, last_login, is_active)
books (book_id, isbn, title, author, genre, publication_year, price, stock_quantity, store_id)
stores (store_id, name, address, city, country, manager_phone, opening_date)

* Принципы построения:
Вертикальное шардирование (разбивка по доменам) — сначала делим таблицы по смысловым кластерам, чтобы разгрузить общее хранилище и выделить ресурсы под специфичные нагрузки.

* Горизонтальное шардирование (разбивка по ключам) — самые тяжелые таблицы дополнительно дробим по хешу ключа, чтобы распределить данные по множеству физических узлов и обеспечить бесконечное масштабирование записи.
